# 耐量子計算機暗号（PQC） 2026

量子コンピュータの登場により、現在広く使われている RSA や ECC（楕円曲線暗号）は将来的に解読される運命にあります。**「今すでに収集され、後で解読される」Harvest Now Decrypt Later 攻撃は進行中**です。NIST FIPS 203/204/205 が確定した 2026 年、移行はもはやオプションではなく、緊急の技術的責務です。

## Section 01 — 概要・なぜ今 PQC が必要なのか

**耐量子計算機暗号（Post-Quantum Cryptography: PQC）**とは、既存の公開鍵暗号（RSA・ECC等）が実用的な量子コンピュータによって数時間で解読されるリスクに対抗するために設計された、新しい数学的問題に基づく暗号アルゴリズム群です。

### 現在の暗号が「なぜ」破られるのか

| 現在の暗号 | 安全性の数学的根拠 | 量子アルゴリズム | 量子による影響 |
| :--- | :--- | :--- | :--- |
| **RSA-2048/4096** | 巨大な数の素因数分解 | Shorのアルゴリズム | **完全に解読可能** (数時間〜) |
| **ECC (P-256 / Ed25519)** | 楕円曲線上の離散対数問題 | Shorのアルゴリズム | **完全に解読可能** (RSAより速い) |
| **AES-256 (対称鍵)** | 鍵空間の広さ | Groverのアルゴリズム | **安全性半減** (128bit相当) |
| **SHA-256/384 (ハッシュ)** | 一方向衝突耐性 | Grover/他 | **影響小** (384/512への移行推奨) |

> 🚨 **重要な誤解の解消**
> 「量子コンピュータが完成してから考えればいい」は間違いです。暗号の寿命は**「データの秘匿必要期間 ＋ システム移行期間」**で決まるからです。

## Section 02 — Harvest Now, Decrypt Later (HNDL) 攻撃

国家レベルの攻撃者は、**現在すでに暗号化されたデータを大量に収集・保存**しています。将来、強力な量子コンピュータが完成した瞬間に、これらを一斉に解読（遡及的解読）することを目論んでいます。

### HNDL 攻撃の影響を最も受けるデータ
- **国家機密・防衛データ**: 数十年にわたる秘匿が必要。
- **個人の医療・ゲノム情報**: 一生涯変わることのない機密。
- **知的財産 (IP)・研究開発データ**: 企業の競争優位性を左右する。
- **長期的な金融取引・契約記録**: 過去の署名の真正性が問われる。

これらのデータを取り扱うシステムでは、**「今日」暗号化するデータが将来解読されるリスク**を考慮し、PQCへの即時移行が求められています。

## Section 03 — NIST PQC 標準アルゴリズム（2024年8月確定）

長年の選考プロセスの末、NIST（米国国立標準技術研究所）は2024年8月13日に以下の3つの標準を正式に公開しました。

### 1. FIPS 203: ML-KEM (旧称: CRYSTALS-Kyber)
- **用途**: 鍵交換 / TLS セッションキーの共有。
- **特徴**: 格子暗号（ML-LWE）ベース。既存暗号に近い性能。
- **推奨**: セキュリティレベル2 (Kyber-768相当)。

### 2. FIPS 204: ML-DSA (旧称: CRYSTALS-Dilithium)
- **用途**: デジタル署名 / アイデンティティ認証。
- **特徴**: 格子暗号（ML-SIS/ML-LWE）ベース。
- **推奨**: 高速な署名生成・検証。

### 3. FIPS 205: SLH-DSA (旧称: SPHINCS+)
- **用途**: ハッシュベースのデジタル署名。
- **特徴**: 格子暗号に予期せぬ欠陥が見つかった際のバックアップ。
- **注意**: 署名サイズが大きく、処理は低速。

## Section 04 — グローバル移行タイムライン ─ 2035年へのカウントダウン

主要な規制当局は、2035年を完全移行の最終期限とするロードマップを提示しています。

### 🇺🇸 米国 (CNSA 2.0 / OMB)
- **2025年**: ソフトウェア・クラウドサービスの署名にPQCを優先採用。
- **2026年**: VPN・ルーター等のネットワーク機器の移行開始。
- **2030年**: レガシーシステム（RSA/ECC）の段階的廃止。
- **2035年**: 国家安全保障システム（NSS）での旧暗号使用を**完全禁止**。

### 🇪🇺 欧州 (ENISA) / 🇯🇵 日本 (CRYPTREC)
- 2026年末までに国家レベルのPQC移行ロードマップを策定。
- ハイブリッドモード（既存暗号 + PQC）を移行期の推奨構成とする。

## Section 05 — Crypto-Agility (暗号の俊敏性) ─ 設計の要諦

PQCへの移行において最も重要なのは、特定のアルゴリズムに依存しない**「Crypto-Agility（暗号の俊敏性）」**を備えた設計です。

### ❌ 悪い設計（ハードコード）
```typescript
// RSA-2048に依存。アルゴリズム変更にはコード修正が必要。
const key = crypto.generateKeyPairSync('rsa', { modulusLength: 2048 });
```

### ✅ 良い設計（抽象化・設定ベース）
```typescript
// 設定ファイルから読み込む。コード変更なしに ML-KEM 等へ切り替え可能。
const algoConfig = loadConfig('crypto.yml');
const key = crypto.generateKeyPairSync(algoConfig.kex_algorithm, algoConfig.params);
```

**Crypto-Agilityの3原則:**
1. 暗号アルゴリズム名や鍵サイズをコードにハードコードしない。
2. 抽象化された暗号インターフェース（DI等）を通じて操作を行う。
3. セキュリティポリシーを中央管理し、動的に反映可能にする。

## Section 06 — ハイブリッドアプローチ ─ 安全な移行の架け橋

PQCは比較的新しい数学的問題に基づいており、将来的に未知の攻撃手法が発見されるリスクがゼロではありません。そのため、**「既存の従来暗号 + 新しいPQC」**を同時に適用する**ハイブリッド方式**が2026年現在のベストプラクティスです。

### ハイブリッド TLS 1.3 の仕組み
`X25519` (従来) と `ML-KEM-768` (PQC) を組み合わせ、それぞれの共有秘密を結合して最終的な鍵を導出します。
- **メリット**: 両方のアルゴリズムが同時に破られない限り、通信の安全性が保たれる。
- **構成例**: Google Chrome や AWS KMS で既にサポート開始。

## Section 07 — 実践：PQC移行 4ステップ

### Step 01 — 暗号資産インベントリ (CBOM) の作成
どこで、どの暗号が使われているかを全て洗い出します。
- **スキャン対象**: SSL/TLS証明書、VPN構成、SSHキー、コード署名、JWTトークン、APIシークレット。

### Step 02 — Crypto-Agility 基盤の構築
新規開発プロジェクトでは、最初からアルゴリズムを動的に切り替えられる設計を採用します。

### Step 03 — ハイブリッドモードのパイロットテスト
テスト環境でPQCハイブリッド構成を導入し、性能影響（ハンドシェイク遅延、CPU負荷、証明書サイズの増加）を測定します。

### Step 04 — デュアルスタック PKI の構築
従来暗号（ECC）とPQC（ML-DSA）の両方の証明書を発行・検証できる環境を構築し、段階的な切り替え準備を整えます。
