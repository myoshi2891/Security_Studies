# 耐量子計算機暗号（PQC）2026 完全ガイド

量子コンピュータの登場により、現在広く使われている RSA や ECC（楕円曲線暗号）は将来的に解読される運命にあります。**「今すでに収集され、後で解読される」Harvest Now Decrypt Later 攻撃は進行中**です。NIST FIPS 203/204/205 が確定した 2026 年、移行は義務ではなく緊急事項です。

## Section 01 — 概要・緊急性

**耐量子計算機暗号（PQC）**とは、現在広く使われているRSA・ECC（楕円曲線暗号）が量子コンピュータによって解読されることを見越して設計された、格子ベース・ハッシュベース・符号ベース等の新しい数学的問題に基づく暗号アルゴリズム群です。

### 現在の暗号が「なぜ」量子コンピュータに破られるのか

| 現在の暗号 | 安全性の根拠 | 量子アルゴリズム | 突破速度 |
| :--- | :--- | :--- | :--- |
| **RSA-2048** | 素因数分解の困難性 | Shorのアルゴリズム | 指数関数的短縮（数時間〜） |
| **ECC (P-256等)** | 離散対数問題 | Shorのアルゴリズム | RSA同様に破られる |
| **AES-256 (対称)** | 鍵空間の広さ | Groverのアルゴリズム | セキュリティ半減（128bit相当） |
| **SHA-256 (ハッシュ)** | 一方向性 | Groverの影響小 | SHA-384/512で十分対応可能 |

> **重要な認識**: 量子コンピュータは「まだない」が、問題はすでに起きています。国家レベルの攻撃者はすでに暗号化されたデータを大量に記録・保存しており、将来量子コンピュータが完成した瞬間にそれらをすべて解読しようとしています。これが **Harvest Now, Decrypt Later (HNDL)** 攻撃です。

---

## Section 02 — HNDL 攻撃の深刻さ

どのシステムが優先的に脅威にさらされるかを理解し、移行の優先順位を決定する必要があります。

- **政府・国防・外交**: 国家機密が解読されれば安全保障への影響は甚大。緊急度：**極めて高い**。
- **医療記録・ゲノムデータ**: 生涯〜数十年間の機密保持が必要。緊急度：**極めて高い**。
- **金融取引・決済インフラ**: 過去取引の否認が可能になるリスク。緊急度：**極めて高い**。
- **知的財産・R&Dデータ**: 競合他社への流出。緊急度：**高い**。

## Section 03 — NIST 標準アルゴリズム（2024年8月確定）

長年の選考プロセスを経て、2024年8月13日に正式公開された 3 つの標準です。

1.  **FIPS 203: ML-KEM** (旧称: CRYSTALS-Kyber)
    - 鍵交換 / TLS 用。格子暗号ベース。
    - 推奨: **ML-KEM-768**。
2.  **FIPS 204: ML-DSA** (旧称: CRYSTALS-Dilithium)
    - デジタル署名 / 認証用。高速。
    - 推奨: **ML-DSA-65**。
3.  **FIPS 205: SLH-DSA** (旧称: SPHINCS+)
    - ハッシュベース署名。格子暗号に対するバックアップ。
    - 署名サイズ大、処理低速だが安全性根拠が堅固。

## Section 04 — グローバル移行タイムライン

- **2025年**: ソフトウェア・クラウドの署名にPQC採用開始。
- **2026年 (現在)**: VPN・ネットワーク機器の移行開始。ハイブリッドモードでの段階的移行期。
- **2030年**: レガシーシステム（RSA/ECC）の段階的廃止開始。
- **2035年**: 米国国家安全保障システム（NSS）で RSA/ECC の使用を完全禁止。

---

## Section 05 — Crypto-Agility（暗号の俊敏性）

Crypto-Agility とは、暗号アルゴリズムをシステムの他の部分から分離し、**コードを変更せずに設定のみで暗号方式を切り替えられる設計思想**です。

### 安全な設計のポイント
- アルゴリズム名や鍵サイズをハードコードしない。
- 抽象インターフェースを通じて暗号操作を実行する。
- 暗号プロバイダーを DI（依存性注入）で交換可能にする。

---

## Section 06 — ハイブリッドアプローチ

PQCには未知の数学的欠陥が発見されるリスクがあるため、**「旧暗号 + PQC を同時適用」**するハイブリッドアプローチが現実的なベストプラクティスです。

### ハイブリッド TLS 1.3 の例
`X25519` (既存) + `ML-KEM-768` (PQC) を組み合わせることで、「両方同時に破られない限り安全」という設計を実現します。

## Section 07 — 実践 4 ステップ

1.  **暗号資産インベントリ (CBOM) の作成**: どこで公開鍵暗号が使われているかリスト化。
2.  **Crypto-Agility 基盤の構築**: アルゴリズムの設定ファイル管理への移行。
3.  **ハイブリッドモードのパイロットテスト**: TLS ハンドシェイク性能の測定。
4.  **デュアルスタック PKI 構築**: PQC と既存 ECC 両方の証明書を発行・検証できる体制。
