# ソフトウェアサプライチェーンセキュリティ 2026

**現代のアプリケーションコードのうち、開発者が自ら書く行数はわずか10〜20%と言われています。** 残りはOSS、外部ライブラリ、CI/CDパイプライン、クラウドベースのビルド環境、そしてコンテナイメージで構成されています。この「外部から持ち込まれる要素」の安全性をいかに確保するかが、2026年のソフトウェア開発における最重要課題です。

## Section 01 — 概要・なぜサプライチェーンが重要なのか

なぜ今、すべての開発者が「サプライチェーン（供給網）」を意識しなければならないのでしょうか。

### サプライチェーン攻撃の定義
ソフトウェアサプライチェーン攻撃とは、ターゲットとなるソフトウェアそのものを直接攻撃するのではなく、そのソフトウェアの**開発・ビルド・配布プロセスの「どこか」**に悪意あるコードを混入させる攻撃です。一度成功すれば、そのソフトウェアを使用する数万、数百万のユーザーを一斉に侵害できるため、攻撃者にとって極めて投資対効果（ROI）が高い攻撃手法となっています。

### 2026年の現実
- **OWASP Top 10:2025 新規追加**: 175,000件以上のCVE分析の結果、**A03 (Software Supply Chain Failures)** が新規カテゴリとして追加されました。
- **インパクト最高カテゴリ**: 発生頻度は他のカテゴリより低いものの、一度発生した際の「平均的なインパクトスコア」が全10カテゴリ中で最高（CRITICAL）となっています。
- **法規制の義務化**: 米国大統領令 EO 14028 や EU サイバーレジリエンス法（CRA）により、SBOM（部品表）の提供と脆弱性の管理が、努力目標から**「法的義務」**へと完全にシフトしました。

## Section 02 — 主要な攻撃ベクター ─ どこが狙われるのか

攻撃者がサプライチェーンを狙う際の、2026年現在の3つの主要なアプローチを理解しましょう。

### 1. 依存関係の汚染 (Dependency Poisoning)
- **Dependency Confusion**: 内部専用パッケージと同名の、より高いバージョンを持つマルウェア入りパッケージを公式レジストリ（npm/PyPI等）に登録し、ビルド時に自動的にマルウェア側を優先してインストールさせる。
- **Typosquatting**: `lodash` を `lodsh` と打ち間違えるのを待つ攻撃。
- **Slopsquatting (2026年新脅威)**: AIコーディングアシスタントが「ハルシネーション（幻覚）」で提案した実在しないパッケージ名を逆手に取り、攻撃者がその名前でマルウェアを先回り登録しておく。

### 2. CI/CDパイプラインの侵害 (CI/CD Hijacking)
- **SolarWinds型攻撃**: ソースコードは安全でも、ビルドを行うサーバー（GitHub Actions, GitLab CI等）が侵害され、バイナリ生成時にバックドアが埋め込まれます。
- **Secret Leakage**: パイプライン内で使用されるクラウドプロバイダーのAPIキーやPAT（個人アクセストークン）が漏洩し、そこから組織全体へ侵害が広がります。

### 3. メンテナーの侵害 (Maintainer Compromise)
- **ソーシャルエンジニアリング**: 2024年の **XZ Utils 事件**のように、攻撃者が数年かけてOSSプロジェクトに貢献し、信頼を得てメンテナー権限を取得した後にバックドアを注入する。
- **アカウント乗っ取り**: MFAを設定していないOSSメンテナーの認証情報を盗み、正当なアップデートとしてマルウェアを配信する。

## Section 03 — SBOM (Software Bill of Materials) ─ 「成分表示」の義務化

### SBOMとは
**SBOM（ソフトウェア部品表）**とは、食品の「原材料表示ラベル」のソフトウェア版です。アプリケーションを構成するすべてのコンポーネント、OSSライブラリ、バージョン、ライセンス、および依存関係のツリーを機械可読な形式で記録したリストです。

### SBOMの2大標準フォーマット (2026年時点)

#### 1. CycloneDX (OWASP推奨) — ★主流
- OWASPが開発・管理する軽量でセキュリティに特化したフォーマット。
- VEX（脆弱性悪用可能性）の直接埋め込みに対応。
- 2026年時点でエンタープライズおよびツールベンダー間で最も広く普及。

#### 2. SPDX (Linux Foundation) — 国際標準
- ISO/IEC 5962:2021として国際標準化済み。
- ライセンスコンプライアンス管理に強みを持つ。
- 大規模なエコシステム（Linux等）での採用が多い。

### 実践：SBOMの生成と活用

```bash
# ━━ Node.js プロジェクトで CycloneDX SBOM を生成 ━━
npm install -g @cyclonedx/cyclonedx-npm
cyclonedx-npm --output-format JSON --output-file sbom.json

# ━━ コンテナイメージ（Trivy使用）から SBOM を抽出 ━━
trivy image --format cyclonedx --output sbom-image.json myapp:v2.1.0

# ━━ 生成したSBOMを基に脆弱性（CVE）を高速スキャン（Grype使用） ━━
grype sbom:./sbom.json
```

## Section 04 — SLSA (スルサ) ─ ビルドの完全性を証明する

### SLSAとは
SBOMが「何が入っているか（成分）」を記録するのに対し、**SLSA (Supply-chain Levels for Software Artifacts)** は「**どのように作られたか（プロセスの完全性）**」を暗号学的に証明するためのフレームワークです。

### SLSA v1.0 のレベル要件

| レベル | 要件の概要 | 防げる攻撃の例 |
| :--- | :--- | :--- |
| **Level 1** | ビルドプロセスをスクリプト化し、来歴（Provenance）を生成。 | 偶発的なビルドミス、再現不可能なビルド。 |
| **Level 2** | ビルドサービスが来歴に自動署名。ソース管理の利用。 | ビルド後のバイナリ改ざん。 |
| **Level 3** | 分離されたビルド環境。ビルドシステムが侵害されても来歴を偽装できない仕組み。 | 侵害されたビルドサーバーによるコード注入。 |

## Section 05 — VEX (Vulnerability Exploitability eXchange) ─ ノイズの排除

SBOMを導入すると、一つのアプリケーションから数百件の「脆弱性（CVE）」が報告されることがあります。しかし、その多くは「ライブラリには含まれるが、自社のアプリではその機能を使っていない」ため、実際には悪用不可能な**「ノイズ」**です。

**VEX** は、この脆弱性が実際に製品に影響するかどうかを伝えるステータス情報です。
- **Affected**: 悪用可能。すぐに対策が必要。
- **Not Affected**: 脆弱なコードは含まれるが、当製品の構成では実行されない。
- **Fixed**: 既に修正済み。
- **Under Investigation**: 現在調査中。

## Section 06 — Cosign & Sigstore ─ 「署名」の民主化

従来、コード署名には複雑な秘密鍵（GPGキー等）の管理が必要でした。**Sigstore** は、これを「鍵なし（Keyless）」で、かつクラウドのアイデンティティ（OIDC）を利用して行えるようにしました。

### Cosign による署名と検証の実践例

```bash
# ━━ 署名 (GitHub Actions 等のCI環境で実行) ━━
# 秘密鍵を管理せず、GitHubのアイデンティティでコンテナイメージに署名
cosign sign --yes ghcr.io/my-org/my-app:latest

# ━━ 検証 (デプロイ前やKubernetes内での検証) ━━
# 正しいGitHubワークフローから署名されたものかを確認
cosign verify ghcr.io/my-org/my-app:latest \
  --certificate-oidc-issuer https://token.actions.githubusercontent.com \
  --certificate-identity https://github.com/my-org/my-repo/.github/workflows/release.yml@refs/heads/main
```

## Section 07 — EU サイバーレジリエンス法 (CRA) ─ 法的法的責任

2026年、セキュリティは「できればやる」から「やらなければ法的責任を問われる」時代へ。

- **脆弱性報告の義務**: 2026年9月11日から、重大なインシデントや悪用された脆弱性を24時間以内にENISA（欧州サイバーセキュリティ機関）へ報告する義務が開始。
- **SBOM提供の義務**: デジタル製品をEU域内で販売する場合、SBOMの維持と提供が必須条件。
- **ペナルティ**: 最大1,500万ユーロ、または年間売上高の2.5%のいずれか高い方の罰金。

## Section 09 — 実践：サプライチェーンセキュリティ・チェックリスト

開発現場でまず優先すべきチェックリストです。

- [ ] **CRITICAL**: 依存関係ロックファイル (`package-lock.json`, `bun.lock` 等) を必ずコミットし、ビルド時に `npm ci` 等で固定している。
- [ ] **HIGH**: CI/CDパイプライン（GitHub Actions 等）で `permissions` を最小権限（`contents: read` 等）に設定している。
- [ ] **HIGH**: コンテナイメージのビルド時に SBOM (CycloneDX) を自動生成し、成果物とセットで保存している。
- [ ] **MEDIUM**: `cosign` を使用し、イメージやバイナリが正当なCIプロセスから署名されているか検証している。
- [ ] **MEDIUM**: 未使用の依存関係や、長期間更新されていないライブラリを定期的に削除している。

## Section 10 — 総括 ─ セキュリティは「連鎖全体の強度」で決まる

サプライチェーンの強度は、**最も弱いリンク**で決まります。
どれだけ自社のコードが安全でも、依存するOSSライブラリ、CI/CDパイプライン、コンテナイメージのいずれかに脆弱性があれば、それがそのまま組織のリスクになります。
SBOMで「何が入っているか」を可視化し、SLSAで「どのように作られたか」を証明し、Cosignで「誰が署名したか」を検証する ── この三位一体が、2026年のソフトウェア開発における核心です。
