# OWASP Top 10:2025 完全解説

**OWASP Top 10** は、世界中のセキュリティ専門家が収集した175,000件以上のCVEデータと実際の侵害事例に基づき、*Webアプリケーションにおける最も重大なセキュリティリスク*をランキング化したものです。2025年版は単なる順位の入れ替えではなく、**設計思想の根本的な変革**を反映しています。

## 概要 — 2025年版の全体像と構造的再編

2025年版では、旧来の「個別のコードバグ」中心の視点から、*クラウドネイティブ・AI・サプライチェーン*の現実を反映した包括的なリスク管理へとシフトしました。特に「サプライチェーン（A03）」と「例外処理の不備（A10）」の新規追加は、現代の開発者が直面する新たな脅威を象徴しています。

### 全10カテゴリ一覧と推移

| 順位 | カテゴリ | 状態 | 変化のポイント |
| :--- | :--- | :--- | :--- |
| **A01** | **Broken Access Control** | 1位維持 | SSRF（サーバー側リクエスト偽造）を新たに統合。 |
| **A02** | **Security Misconfiguration** | 5位→2位 ⬆️ | IaC普及による設定ミスリスクの増大。 |
| **A03** | **Software Supply Chain Failures** | 🆕 新規追加 | **最高インパクト。** OSSやCI/CDの侵害をカバー。 |
| **A04** | **Cryptographic Failures** | 2位→4位 ↘️ | TLS普及により順位は下がったが依然重要。 |
| **A05** | **Injection** | 3位→5位 ↘️ | ORMの普及により減少傾向。クロスサイトスクリプティング（XSS）を包含。 |
| **A06** | **Insecure Design** | 4位→6位 ↘️ | 「実装バグ」ではなく「設計の欠陥」への注力。 |
| **A07** | **Authentication Failures** | 変動なし | アイデンティティ攻撃の激化。AIによる突破リスク。 |
| **A08** | **Software and Data Integrity Failures** | 変動なし | 更新プロセスやデシリアライズの不備。 |
| **A09** | **Security Logging and Alerting Failures** | 変動なし | 記録だけでなく「プロアクティブなアラート」を重視。 |
| **A10** | **Mishandling of Exceptional Conditions** | 🆕 新規追加 | エラー処理の不備による情報漏洩やフェイルオープン。 |

## A01:2025 — Broken Access Control（アクセス制御の不備）

ユーザーが自分の権限を超えてデータや機能にアクセスできてしまう問題。SSRFも統合されました。

### ⚠ 攻撃シナリオ
- **IDOR**: URLの注文ID（`?id=1001`）を書き換えて、他人の個人情報を閲覧する。
- **SSRF**: 画像のURL指定機能に `http://169.254.169.254/latest/meta-data/` を入力させ、AWSの認証情報を盗む。

### ✅ 対策
- **Deny by Default**: 明示的に許可されていないアクセスはすべて拒否する。
- **所有権の検証**: データベースクエリ時に必ず「このデータはログインユーザーのものか」を確認する。
- **UUIDの使用**: 推測可能な連番ID（1, 2, 3...）を廃止し、予測不可能なUUIDを採用する。

## A02:2025 — Security Misconfiguration（セキュリティ設定の不備）

デフォルト設定の放置、不必要な機能の有効化、詳細すぎるエラーメッセージなど。

### ✅ 対策
- **IaCスキャン**: CI/CDパイプラインで Terraform や CloudFormation の設定ミスを自動検知する（Trivy等）。
- **デフォルト認証情報の変更**: admin/admin などの初期設定を物理的に廃止する。
- **環境分離**: `NODE_ENV=production` を徹底し、本番環境でデバッグ情報やスタックトレースを表示させない。

---

## A03:2025 — Software Supply Chain Failures（サプライチェーンの欠如） ─ 🆕

他者のコード（OSSライブラリ、コンテナイメージ、ビルド済みバイナリ）に起因する脆弱性。

### ✅ 対策
- **SBOMの生成**: `CycloneDX` 形式等で全依存関係を可視化・追跡する。
- **ロックファイルの徹底**: `package-lock.json` 等をコミットし、ハッシュ値による整合性チェックを行う。
- **Slopsquatting対策**: AIがハルシネーション（幻覚）で提案した実在しないパッケージ名を、公式レジストリで必ず照合する。

---

## A04:2025 — Cryptographic Failures（暗号化の失敗）

機密データの不適切な保護。以前の「機密データの露出」から、暗号技術自体の不備へと焦点が移りました。

### ✅ 対策
- **bcrypt/Argon2id**: パスワード保存にはSHA-256ではなく、ソルト付きの低速な専用ハッシュ関数を使用する。
- **暗号化の廃止**: MD5, SHA-1, DES などの脆弱なアルゴリズムをシステムから完全に排除する。
- **PQCへの準備**: 耐量子計算機暗号への移行を視野に入れた設計（Crypto-Agility）を開始する。

---

## A05:2025 — Injection（インジェクション）

ユーザー入力をそのまま命令として実行してしまう問題。SQLインジェクションやXSSを含みます。

### ✅ 対策
- **プレースホルダー**: 文字列結合によるクエリ生成を厳禁し、パラメータ化されたクエリ（Prepared Statements）を使用する。
- **出力エンコーディング**: ブラウザへ出力する際は、常にコンテキストに応じたサニタイズ（DOMPurify等）を行う。

---

## A06:2025 — Insecure Design（不安全な設計）

実装以前の、根本的な設計ミス。ビジネスロジックの欠陥。

### ✅ 対策
- **脅威モデリング**: 設計フェーズで STRIDE 等の手法を用いて、想定される攻撃を洗い出す。
- **セキュアな設計原則**: 最小権限原則やフェイルクローズ設計をアーキテクチャに組み込む。

---

## A07:2025 — Authentication Failures（認証の不備）

セッション管理の不備、ブルートフォースへの耐性欠如、多要素認証（MFA）の未実装。

### ✅ 対策
- **パスキー (FIDO2)**: SMS OTPを廃止し、フィッシング耐性のある認証へ移行する。
- **レート制限**: ログイン試行回数に厳格な制限を設け、アカウントロック機能を実装する。

---

## A08:2025 — Software and Data Integrity Failures（完全性の欠如）

署名されていないコードの更新、安全でないデシリアライズ。SolarWinds事件のような侵害を対象とします。

### ✅ 対策
- **コード署名**: 開発・ビルド・配布の各工程でデジタル署名を行い、改ざんを検知可能にする。
- **署名検証**: 外部から取得したライブラリやデータを使用する前に、署名を必ず検証する。

---

## A09:2025 — Security Logging and Alerting Failures（ログ・アラートの不備）

攻撃が発生しても、それに気づけない状態。

### ✅ 対策
- **構造化ログ**: JSON形式等で、検索・解析しやすいログを出力する。
- **プロアクティブな通知**: ログイン失敗の急増や、特権操作の発生をリアルタイムでセキュリティチームに通知する。

---

## A10:2025 — Mishandling of Exceptional Conditions（例外状況の不適切な処理） ─ 🆕

エラー処理の不備による情報漏洩や、システム停止。

### ✅ 対策
- **フェイルクローズ**: 認証システム等でエラーが発生した際は、デフォルトでアクセスを拒否する設計にする。
- **詳細ログ・汎用エラー**: システムの詳細はサーバーログに記録し、ユーザーには汎用的なメッセージのみを返す。

## 総括 — 2026年の開発者に求められること

OWASP Top 10 は単なるチェックリストではありません。2026年、セキュリティは開発者にとって**「正しく動作するコード」の定義そのもの**となりました。
本ガイドの各項目を一つずつエンジニアリングプラクティスに組み込み、継続的な改善サイクルを回し続けることが、組織のセキュリティレジリエンス（回復力）を高める唯一の道です。
