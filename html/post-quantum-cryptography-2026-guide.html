Section 01 — Overview

PQCとは何か
なぜ「今すぐ」始めなければならないのか

耐量子計算機暗号（Post-Quantum Cryptography）は、量子コンピュータでも解読できない次世代暗号アルゴリズム群。2026年は移行の「始まり」ではなく「すでに遅れている」段階です。

耐量子計算機暗号（PQC）とは、現在広く使われているRSA・ECC（楕円曲線暗号）が 量子コンピュータによって解読されることを見越して設計された、 格子ベース・ハッシュベース・符号ベース等の新しい数学的問題に基づく暗号アルゴリズム群です。 NISTが2024年8月に3つの標準（FIPS 203/204/205）を正式公開し、 世界の政府・企業が移行を始めています。

現在の暗号が「なぜ」量子コンピュータに破られるのか

現在の暗号安全性の根拠（数学的困難性）量子アルゴリズム突破速度
RSA-2048 大きな数の素因数分解が困難 Shorのアルゴリズム 指数関数的短縮（数時間〜）
ECC（P-256等） 楕円曲線上の離散対数問題 Shorのアルゴリズム RSA同様に破られる
DH鍵交換 離散対数問題 Shorのアルゴリズム 同様に無効化
AES-256（対称） 鍵空間の広さ Groverのアルゴリズム セキュリティ半減（128bit相当）→ 鍵長倍増で対応可能
SHA-256/SHA-3（ハッシュ） 一方向性・衝突耐性 Groverの影響小 SHA-384/512で十分対応可能
🚨
重要な認識：量子コンピュータは「まだない」が問題はすでに起きている Shorのアルゴリズムを実行できる大規模量子コンピュータ（CRQC）は現時点で存在しません。 しかし国家レベルの攻撃者はすでに暗号化されたデータを大量に記録・保存しています。 10年後にCRQCが完成した瞬間、今日収集したデータがすべて解読されます。 これが「Harvest Now, Decrypt Later（今収集し、後で解読）」攻撃の本質です。

Section 02 — 最重要脅威

Harvest Now, Decrypt Later
― すでに進行中の攻撃を理解する

HNDL攻撃は未来の脅威ではない。国家支援型攻撃者は今この瞬間もデータを収集している。長期秘匿が必要なすべてのデータが今すぐ危険にさらされている。

CRITICAL — 進行中 国家レベル攻撃者
⚠ HARVEST NOW, DECRYPT LATER 攻撃フロー
Phase 1（現在 — 継続中）: 国家支援型攻撃者が政府・金融・医療・インフラの暗号化通信を VPN・TLSセッション含めて大量に傍受・保存。 現在の鍵では解読できないため、ストレージに蓄積し続ける。

Phase 2（5〜15年後 — Q-Day）: 暗号学的に有意な量子コンピュータ（CRQC）が実用化される。 ShorのアルゴリズムをCRQCで実行し、 Phase 1で収集したすべての過去データを一括解読。

被害の深刻さ: 今日収集されているデータには 10〜30年の機密保持が必要な情報（外交電報・医療記録・知的財産・金融取引）が含まれる。 これらは量子コンピュータが実用化される頃には「過去の通信」だが、 被害は現実かつ深刻。

Section 03 — NIST 標準アルゴリズム

NIST FIPS 203 / 204 / 205
― 2024年8月正式確定の3標準

長年の選考プロセスを経て2024年8月13日に正式公開。これが世界の移行基準となるPQC標準アルゴリズム。用途によって使い分けが必要。

2024年8月 正式確定 NIST一次ソース
FIPS 203
ML-KEM
(旧称: CRYSTALS-Kyber)
鍵カプセル化メカニズム（KEM）。 TLSハンドシェイク等でセッション鍵を安全に交換するために使用。 格子暗号（Module-Lattice）ベース。 3つの強度設定が選択可能: ML-KEM-512（128bit安全性）、 ML-KEM-768（192bit・推奨）、 ML-KEM-1024（256bit）。
鍵交換 / TLS
FIPS 204
ML-DSA
(旧称: CRYSTALS-Dilithium)
デジタル署名アルゴリズム。 コード署名・証明書・ドキュメント署名・認証に使用。 格子暗号ベースで高速。 3つの強度設定: ML-DSA-44（128bit）、 ML-DSA-65（192bit・推奨）、 ML-DSA-87（256bit）。
デジタル署名 / 認証
FIPS 205
SLH-DSA
(旧称: SPHINCS+)
ハッシュベースデジタル署名。 格子暗号に対するバックアップとして位置づけ。 格子暗号の数学的欠陥発見に備えた保険的役割。 署名サイズが大きく処理が遅いがセキュリティ根拠が堅固。 長期保存データの署名に適切。
署名（バックアップ用途）
⚠️
候補4: FN-DSA（FALCON）について FALCON（FIPS 206として標準化予定）も署名アルゴリズムとして選定されています。 ML-DSAより署名サイズが小さく、IoT等リソース制約環境に向いています。 ただし実装の難易度が高いため、まずML-DSAでの実装を推奨します。

Section 05 — 設計原則

Crypto-Agility（暗号の俊敏性）
― アルゴリズムをハードコードしない設計

特定の暗号アルゴリズムをコードに埋め込まず、設定変更だけで切り替えられる設計。PQC移行を成功させる唯一の持続可能なアーキテクチャ。

ℹ Crypto-Agility とは
Crypto-Agilityとは、暗号アルゴリズムをシステムの他の部分から分離し、 コードを変更せずに設定・モジュール交換だけで暗号方式を変更できる設計思想です。 PQCが特定のアルゴリズムへの移行だけでなく、 「将来のアルゴリズム変更にも対応できる基盤を作る」ことが本当の目標です。 実際、PQC標準自体も将来的に更新される可能性があります（格子暗号に未知の数学的欠陥が発見された場合等）。
❌ Crypto-Agility なし（危険な設計）
  • アルゴリズム名（"RSA"/"AES-256"）をコードにハードコード
  • 鍵サイズを定数で埋め込む（keySize = 2048）
  • 暗号ライブラリを直接importし全体に散在
  • アルゴリズム変更 = 全コードの修正が必要
  • テスト・検証のたびに大規模リファクタリング
✅ Crypto-Agility あり（安全な設計）
  • 暗号アルゴリズムを設定ファイル（crypto.yml）で管理
  • 抽象インターフェースを通じて暗号操作を実行
  • 暗号プロバイダーをDI（依存性注入）で交換可能に
  • アルゴリズム変更 = 設定ファイル1行の変更のみ
  • ハイブリッドモード（旧+新の同時利用）に対応
crypto-agility.ts — アルゴリズムをハードコードしない設計 
// ━━ ✅ Crypto-Agility: アルゴリズムを設定から動的に読み込む設計 ━━

// 暗号設定（コードには一切ハードコードしない）
interface CryptoConfig {
  kex: 'X25519' | 'ML-KEM-768' | 'X25519+ML-KEM-768'; // 鍵交換
  sig: 'Ed25519' | 'ML-DSA-65' | 'Ed25519+ML-DSA-65'; // 署名
  sym: 'AES-256-GCM' | 'ChaCha20-Poly1305';            // 対称暗号
}

// crypto.yml から動的に読み込み（コード変更なし）
const config: CryptoConfig = loadConfig('./crypto.yml');

// 暗号操作の抽象インターフェース（実装から分離）
interface CryptoProvider {
  generateKeyPair(): Promise<{ publicKey: Uint8Array; privateKey: Uint8Array }>;
  sign(data: Uint8Array, privateKey: Uint8Array): Promise;
  verify(data: Uint8Array, sig: Uint8Array, publicKey: Uint8Array): Promise;
  encapsulate(publicKey: Uint8Array): Promise<{ ciphertext: Uint8Array; sharedSecret: Uint8Array }>;
  decapsulate(ciphertext: Uint8Array, privateKey: Uint8Array): Promise;
}

// ━━ 移行期のハイブリッドプロバイダー ━━
// 旧アルゴリズム(X25519) と 新アルゴリズム(ML-KEM-768) を同時実行
class HybridKEMProvider implements CryptoProvider {
  constructor(
    private classical: CryptoProvider,  // X25519（既存）
    private pqc: CryptoProvider,         // ML-KEM-768（新規）
  ) {}

  async encapsulate(publicKey: Uint8Array) {
    // 両方のアルゴリズムで独立に鍵交換
    const classical = await this.classical.encapsulate(publicKey);
    const pqc       = await this.pqc.encapsulate(publicKey);

    // 共有秘密をXORで結合: 片方が破られても安全
    const sharedSecret = xor(classical.sharedSecret, pqc.sharedSecret);
    const ciphertext   = concat(classical.ciphertext, pqc.ciphertext);
    return { ciphertext, sharedSecret };
  }
  // ... generateKeyPair, decapsulate 等も同様
}

// ━━ 使用側（アルゴリズムに依存しない） ━━
// crypto.yml を変更するだけで X25519 → ML-KEM-768 へ切り替え可能
const provider = createProvider(config); // ← 設定に基づき自動選択
const session = await provider.encapsulate(serverPublicKey);

Section 06 — 移行戦略

ハイブリッドアプローチ
― 安全に移行するための現実的な戦略

旧来の暗号とPQCを「同時に」使用することで、従来型コンピュータと量子コンピュータ両方の攻撃に対して堅牢性を維持しながら段階的に移行する。

ℹ なぜハイブリッドアプローチが必要か
PQCアルゴリズムはまだ歴史が浅く、未知の数学的欠陥が発見されるリスクがゼロではありません。 ハイブリッドアプローチ（旧暗号 + PQC を同時適用）を使えば： PQCが破られても従来暗号が守り、従来暗号が量子で破られてもPQCが守る ── 「両方同時に破られない限り安全」という設計。 これがNISTも推奨する移行期の現実的なベストプラクティスです。

ハイブリッドTLS 1.3 の仕組み（X25519 + ML-KEM-768）

TLS 1.3のハンドシェイクにハイブリッド鍵交換を組み込む場合、 クライアントとサーバーはX25519（既存）とML-KEM-768（PQC）の 両方の鍵交換を独立して実行し、 それぞれの共有秘密をKDF（鍵導出関数）で結合します。

Client → Server:
X25519 ephemeral public key (32 bytes)
ML-KEM-768 encapsulation key (1,184 bytes)
// 合計 ~1,216 bytes のClientHello拡張
Server → Client:
X25519 ephemeral public key (32 bytes)
ML-KEM-768 ciphertext (1,088 bytes)
// 合計 ~1,120 bytes のServerHello
共有秘密 = HKDF(X25519_secret || ML-KEM_secret)
// 片方が破られても他方が保護 — 両方同時に破られない限り安全
nginx-pqc.conf — NGINXでのハイブリッドTLS設定（概念） 
# ━━ OpenSSL 3.x + OQS Provider でハイブリッドTLSを構成する例 ━━
# 前提: OQS-OpenSSL (liboqs) のインストールが必要

# nginx.conf の ssl_conf_command でハイブリッドKEMを指定
server {
    listen 443 ssl;
    ssl_certificate     /etc/ssl/certs/server.pem;
    ssl_certificate_key /etc/ssl/private/server.key;
    
    # ハイブリッド鍵交換グループを指定
    # X25519MLKEM768 = X25519 + ML-KEM-768 のハイブリッド
    ssl_conf_command Groups X25519MLKEM768:X25519:P-256;
    #                ^^^^^^^^^^^^^^^^^^^^
    #                最優先: ハイブリッドKEM（クライアントが対応していれば使用）
    #                フォールバック: X25519, P-256（旧来のブラウザ向け）
    
    ssl_protocols TLSv1.3;
    ssl_prefer_server_ciphers off;
}

# ━━ テスト: ハイブリッドKEMが使われているか確認 ━━
openssl s_client -connect example.com:443 \
  -groups X25519MLKEM768 \
  -tls1_3 2>&1 | grep "Server Temp Key"
# 出力例: Server Temp Key: X25519MLKEM768, 1216 bits
#          ← ハイブリッドKEMが使用されていることを確認

# ━━ Python (oqs-python) での実装例 ━━
# pip install oqs
import oqs

# ML-KEM-768 KEMのテスト
with oqs.KeyEncapsulation("Kyber768") as kem:
    public_key = kem.generate_keypair()
    ciphertext, shared_secret_server = kem.encap_secret(public_key)
    shared_secret_client = kem.decap_secret(ciphertext)
    assert shared_secret_server == shared_secret_client
    print(f"Shared secret: {shared_secret_client.hex()[:32]}...")

01
暗号資産インベントリの作成
まず組織内のすべてのシステムで公開鍵暗号がどこで使われているかをリスト化します。 「暗号インベントリ（Cryptographic Bill of Materials: CBOM）」の作成です。 対象は TLS証明書・VPN・SSH・コード署名・JWTトークン・APIキー・データベース暗号化・S/MIMEメール・ファームウェア署名。
crypto-inventory.sh — 使用中の暗号アルゴリズムをスキャン 
# ━━ TLS証明書のアルゴリズムを確認 ━━
echo | openssl s_client -connect your-domain.com:443 2>/dev/null \
  | openssl x509 -noout -text \
  | grep -E "Public Key Algorithm|Signature Algorithm|Public-Key:"
# 出力例:
# Signature Algorithm: sha256WithRSAEncryption  ← RSA → PQC移行が必要
# Public-Key: (2048 bit)                        ← ML-DSAに置き換え必要

# ━━ TLSハンドシェイクの鍵交換方式を確認 ━━
openssl s_client -connect your-domain.com:443 2>&1 | grep "Server Temp Key"
# 出力例:
# Server Temp Key: ECDH, prime256v1, 256 bits   ← ECC → ML-KEM移行が必要

# ━━ ネットワーク内のTLS証明書を一括スキャン（SSLyze使用） ━━
pip install sslyze
python -m sslyze --regular your-domain.com \
  | grep -E "keyType|curveName|sigHashAlgorithm"

Crypto-Agility基盤の構築
暗号アルゴリズムをハードコードしている箇所を特定し、 抽象レイヤーと設定ファイルを通じた管理に移行します。 この作業はPQCへの移行とは独立して実施でき、 完了すれば将来のアルゴリズム変更コストが劇的に下がります。 新規プロジェクトは最初からCrypto-Agility設計で構築してください。

本番環境への即座の切り替えではなく、テスト環境でX25519 + ML-KEM-768のハイブリッドTLSを構成し、 以下を計測・評価します:
pqc-benchmark.sh — TLSハンドシェイク性能測定 
# ━━ TLSハンドシェイクの遅延比較 ━━

# 従来のTLS（X25519）

# ハイブリッドTLS（X25519 + ML-KEM-768）
time for i in $(seq 1 100); do

# 3. CPU使用率の変化（ML-KEMの鍵生成・カプセル化コスト）
# 4. 既存クライアント（旧ブラウザ等）との後方互換性

ninja && sudo ninja install

総括

「始めた者だけが間に合う」
― PQC移行は今から

量子コンピュータはまだ実用化されていません。 しかし、それが「待っていい理由」ではありません。 移行には5〜10年のリードタイムが必要であり、 Harvest Now Decrypt Later攻撃によって今日の暗号化データはすでに危険にさらされています。 NIST FIPS 203/204/205が確定した今、技術的な根拠は揃っています。 あとは組織として行動を起こすだけです。

今すぐ始めること理由
暗号インベントリ（CBOM）の作成移行すべき対象が分からなければ何も始まらない
Crypto-Agility設計の導入将来のアルゴリズム変更コストを劇的に削減
ML-KEM-768ハイブリッドのテスト環境構築実際の性能影響を測定し、本番移行計画を立てる
CNSA 2.0タイムラインの社内計画への組み込み規制対応の遅れを防ぎ、取締役会への説明責任を果たす
