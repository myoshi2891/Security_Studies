Section 01 — Overview

ソフトウェアサプライチェーンとは何か
なぜ今これほど重要なのか

OWASP Top 10:2025でA03として新規追加。発生頻度は低いがインパクトスコアが全カテゴリ中最高。SolarWinds・Log4j・XZ Utilsが示した教訓。

ソフトウェアサプライチェーンとは、一つのソフトウェアが完成するまでに関わる すべての材料・プロセス・ツール・人の連鎖を指します。 ソースコードだけでなく、依存ライブラリ・ビルドシステム・CI/CDパイプライン・ コンテナイメージ・パッケージレジストリ・クラウドインフラがすべて含まれます。 攻撃者はこの「連鎖の最も弱いリンク」を狙います。

なぜ今サプライチェーン攻撃が急増しているのか

背景要因 具体的な変化 セキュリティへの影響
OSSへの依存度の爆発的増加 現代アプリのコードの80〜90%がOSSライブラリで構成 「他者のコード」の脆弱性が自組織のリスクに直結
CI/CDパイプラインの複雑化 GitHub Actions・ArgoCD等による自動化が普及。ビルド過程が「ブラックボックス」化 パイプライン自体が攻撃対象に（SolarWindsの教訓）
コンテナ・マイクロサービス化 DockerHub等から取得したベースイメージを多数組み合わせて使用 ベースイメージに含まれるCVEが全てのサービスに影響
AIコーディング普及 AIが実在しないパッケージ名を提案する「Slopsquatting」が急増 マルウェア入りパッケージを意図せずインストール
法規制の強化 EU CRA・米国大統領令でSBOM提出が事実上義務化 未対応で法的ペナルティ・調達からの除外リスク

代表的なサプライチェーン攻撃事件（歴史的教訓）

SolarWinds（2020年）
CI/CD侵害
正規ソフトウェアのビルドパイプラインに攻撃者が侵入し、 アップデートにバックドア（SUNBURST）を埋め込み配布。 米国政府機関を含む18,000社以上が影響を受けた。 CI/CDパイプライン自体が攻撃対象になることを世界に示した転換点。
Log4Shell（2021年）
依存関係脆弱性
Javaの人気ライブラリLog4jにRCE（遠隔コード実行）脆弱性。 世界中のほぼすべてのJavaアプリケーションが影響を受けたが、 どのシステムにLog4jが含まれるか把握できていた組織はほとんどなかった。 SBOMの重要性を世界に知らしめた事件。
XZ Utils（2024年）
長期潜伏型
OSSプロジェクトに2年以上かけて信頼を積み上げた攻撃者が メンテナー権限を取得し、バックドアを含むコードをマージ。 Linuxシステムへの大規模な侵害を未然に防いだが、 人間のソーシャルエンジニアリングによるサプライチェーン攻撃の脅威を示した。

Section 02 — Threat Vectors

主要な攻撃ベクター
5つのパターンを知る

サプライチェーン攻撃には複数の侵入経路があります。それぞれのパターンと具体的な攻撃シナリオを理解することが防御の第一歩です。

① 依存関係の汚染（Dependency Confusion / Typosquatting）

Typosquatting: reqeusts（typo）vs requests（正規）のように、スペルミスを誘う名前のマルウェアパッケージを公開レジストリに登録。
Dependency Confusion: 企業の内部パッケージと同名のパッケージをnpm等のパブリックレジストリに高バージョンで登録。パッケージマネージャーが「より新しいバージョン」として自動選択してしまう。

⚠ 攻撃シナリオ
企業が内部レジストリで [email protected] を使用している事実をGitHubから把握した攻撃者が、 npmに [email protected]（マルウェア入り）を登録。 npm install 実行時にパッケージマネージャーが「より新しい版がある」と判断し、 攻撃者のパッケージを自動選択・インストールしてしまう。
② CI/CDパイプラインの侵害

ビルド・テスト・デプロイを自動化するCI/CDパイプライン自体が攻撃対象になります。 SolarWinds事件はこのパターンの典型例です。 パイプラインへの不正アクセスに成功すれば、正規のビルドプロセスにバックドアを混入できます。

侵害手法 説明 対策
シークレットの漏洩 CI/CDのシークレット（APIキー等）がログや環境変数から漏洩 シークレット管理サービス利用 + ログのスクラブ
PIN止めのないAction GitHub Actionsでタグ指定（@v3）を使うと改ざんされたタグに誘導される SHAハッシュでピン止め（@abc123def456）
不要な権限 CI/CDに本番環境への過剰な権限を付与 最小権限・ジョブ単位のOIDCトークン
サードパーティOrbsの汚染 CircleCI Orbsなど再利用可能コンポーネントの改ざん 信頼できるソースのみ使用・バージョン固定
③ メンテナーアカウントの乗っ取り（XZ Utils型）

長期間にわたってOSSプロジェクトに貢献し信頼を積み重ねたあと、 メンテナー権限を取得してバックドアを混入する最も高度な攻撃です。 2024年のXZ Utils事件では約2年間にわたってソーシャルエンジニアリングが行われました。 対策: コードレビューの徹底・多数のメンテナー・プロベナンスの検証が必要。

④ レジストリへの直接攻撃 / ⑤ ビルドツールへの攻撃

npmやPyPI等のパッケージレジストリ自体を侵害して既存の正規パッケージを改ざんするケースや、 コンパイラ・ビルドシステム自体に悪意あるコードを混入する 「コンパイラ攻撃（Ken Thompson's Trust Attack）」も理論的に存在します。 これらに対してはSLSAフレームワークとCosignによる署名検証が最も有効な防御策です。

Section 03 — SBOM

SBOM（ソフトウェア部品表）
─ サプライチェーン可視化の基盤

Log4Shell事件で「このアプリはLog4jを使っているか？」に即答できた組織はほとんどなかった。SBOMはその問題を根本的に解決する。

義務化進行中 米国・EU政府要件
📋
SBOMとは — 一言定義 SBOM（Software Bill of Materials：ソフトウェア部品表）とは、 食品の「原材料表示ラベル」のソフトウェア版です。 アプリケーションを構成するすべてのコンポーネント・ライブラリ・バージョン・ライセンス・ 依存関係（推移的な依存も含む）を機械可読な形式で記録したリストです。

SBOMの2大フォーマット

CycloneDX（OWASP推奨）
  • ▶OWASPが開発・管理する軽量フォーマット
  • ▶セキュリティ用途（脆弱性・ライセンス）に最適化
  • ▶VEX（脆弱性悪用可能性）の直接埋め込みに対応
  • ▶JSON・XML・Protobufで出力可能
  • ▶2026年時点で最も普及しているフォーマット
SPDX（Linux Foundation）
  • ▶Linux Foundationが開発・管理
  • ▶ライセンスコンプライアンスに強み
  • ▶ISO/IEC 5962:2021として国際標準化済み
  • ▶SPDX-TV・JSON・YAML・RDF等で出力
  • ▶米国CISA・EU CRAが参照する標準フォーマット

SBOMの生成方法（実践）

generate-sbom.sh — 各言語のSBOM生成コマンド 
# ━━ Node.js / npm プロジェクト（CycloneDX形式） ━━
npm install -g @cyclonedx/cyclonedx-npm
cyclonedx-npm --output-format JSON --output-file sbom.json
# → sbom.json にすべての依存関係（推移的依存含む）が出力される

# ━━ Python プロジェクト ━━
pip install cyclonedx-bom
cyclonedx-py environment --output sbom-python.json

# ━━ Go プロジェクト ━━
go install github.com/CycloneDX/cyclonedx-gomod/cmd/cyclonedx-gomod@latest
cyclonedx-gomod app -output sbom-go.xml

# ━━ コンテナイメージ（Trivy使用） ━━
trivy image --format cyclonedx --output sbom-image.json myapp:v1.0.0

# ━━ 生成したSBOMで既知CVEをスキャン（Grype使用） ━━
grype sbom:./sbom.json          # 全CVEを検出
grype sbom:./sbom.json --fail-on high  # HIGH以上でCI失敗

# ━━ SBOMの内容確認（人間が読める形式） ━━
# コンポーネント数を確認
cat sbom.json | jq '.components | length'
# 出力例: 423 ← あなたのアプリが間接的に依存するコンポーネント数

Section 04 — SLSA Framework

SLSA（スルサ）
─ ビルドの完全性を暗号学的に証明する

Supply-chain Levels for Software Artifacts。「このビルド成果物が本当に期待されたソースコードから作られたか」を検証するフレームワーク。

🏗️
SLSAとは — SBOMとの違い SBOMが「何が入っているか（部品表）」を記録するのに対し、 SLSAは「どのように作られたか（製造プロセスの証明）」を暗号学的に証明します。 具体的には「来歴（Provenance）」という文書を生成し、 「どのリポジトリの・どのコミットから・どのビルドシステムで・誰が承認して作られたか」を記録します。

SLSA レベル（Level 1〜3）

レベル 要件 何が防げるか 難易度
Level 1 ビルドプロセスをドキュメント化し、来歴（Provenance）を生成する 偶発的なビルドエラーや誤ったビルド設定を発見できる 低（GitHub Actions で自動化可能）
Level 2 ビルドサービスが来歴を自動生成・署名する。コードレビューが必須 手動で改ざんされたビルドを検出できる 中（CI/CDの設定が必要）
Level 3 分離されたビルド環境・監査可能なビルドサービス・来歴への強力な署名 侵害されたビルドシステムやインサイダー脅威も防御 高（高度な設定が必要）

GitHub Actions での SLSA Level 3 実装

.github/workflows/slsa-build.yml — SLSA Level 3 ビルド 
name: SLSA Secured Build Pipeline
on:
  push:
    tags: ['v*']   # リリースタグでのみ実行

permissions:
  id-token: write   # OIDC トークン（Sigstore認証に必要）
  contents: read
  actions: read

jobs:
  # ━━ Step 1: ビルド ━━
  build:
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.hash.outputs.digest }}
    steps:
      - uses: actions/checkout@v4
      - name: Build artifact
        run: |
          npm ci
          npm run build
          tar -czf artifact.tar.gz dist/

      - name: Generate SHA256 hash
        id: hash
        run: |
          DIGEST=$(sha256sum artifact.tar.gz | base64 -w0)
          echo "digest=$DIGEST" >> $GITHUB_OUTPUT

      - uses: actions/upload-artifact@v4
        with:
          name: artifact
          path: artifact.tar.gz

  # ━━ Step 2: SLSA Provenance（来歴）の生成 ━━
  # slsa-github-generator が分離されたビルド環境で来歴を作成・署名
  provenance:
    needs: [build]
    permissions:
      actions: read
      id-token: write
      contents: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/[email protected]
    with:
      base64-subjects: "${{ needs.build.outputs.digest }}"
      # ↑ ビルド成果物のハッシュを渡す → 来歴とビルド成果物が紐付けられる

SLSA 来歴の検証（消費側）

slsa-verify.sh — ダウンロードした成果物の来歴検証 
# slsa-verifier のインストール
go install github.com/slsa-framework/slsa-verifier/v2/cli/slsa-verifier@latest

# ━━ コンテナイメージのSLSA検証 ━━
slsa-verifier verify-image \
  "myapp:v1.2.0" \
  --provenance-path provenance.json \
  --source-uri github.com/my-org/my-repo \
  --source-tag v1.2.0

# ✅ 成功時の出力:
# PASSED: Verified SLSA provenance
# Builder ID: https://github.com/slsa-framework/slsa-github-generator/.github/workflows/...

# Source digest: sha1:a1b2c3d4...

# ━━ バイナリ/アーティファクトのSLSA検証 ━━

Section 05 — VEX

VEX（脆弱性悪用可能性交換）
─ SBOMのノイズを信号に変える

SBOMを導入すると「検出される脆弱性が多すぎて対応できない」問題が発生する。VEXはそのノイズ問題を解決する付随文書。

⚠️
SBOMが生む「アラート疲れ」の問題 SCAツールがSBOMを解析すると、コンポーネントに含まれる全CVEが報告されます。 しかし実際には脆弱なコードが実際に呼び出されているか・悪用可能かどうかは別問題です。 例えば「OpenSSLのX脆弱性」が報告されても、 自分のアプリがその機能を一切使っていなければ実質的なリスクはゼロです。 VEXがなければ、チームは偽陽性の海に溺れてしまいます。

VEXの4つのステータス

ステータス 意味 使用例
Affected この脆弱性は当製品で実際に悪用可能 早急なパッチ適用が必要。ユーザーへの通知も必要
Not Affected 脆弱なコードは含まれるが、当製品では悪用不可能 コンパイラ最適化で除外・設定で無効化・対象機能を未使用
Fixed この脆弱性は既に修正済み パッチ適用済みバージョンを使用中。再評価不要
Under Investigation 現在影響を調査中 脆弱性公開直後。48〜72時間以内にステータスを更新する

Section 06 — Sigstore / Cosign

Cosign & Sigstore
─ キーレス署名で供給元を証明する

「このコンテナイメージは本当に私たちのCIが作ったものか？」を暗号学的に証明するSigstoreエコシステム。2026年のデファクトスタンダード。

🔐
Sigstoreとは — 「コードのLets Encrypt」 Sigstoreは、開発者が秘密鍵を管理せずにソフトウェア成果物に署名できる オープンソースの署名基盤です。 HTTPSの普及に貢献したLet's Encryptと同様の思想で、 ソフトウェア署名を「簡単・無料・広く普及させる」ことを目指しています。 Cosign（コンテナ署名）・Fulcio（CA）・Rekor（透明性ログ）の3コンポーネントで構成。

完全実践チュートリアル（Step by Step）

01
Cosignのインストールと初期設定
install-cosign.sh 
# macOS (Homebrew)
brew install cosign

# Linux (手動インストール)
curl -O -L "https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64"
chmod +x cosign-linux-amd64
sudo mv cosign-linux-amd64 /usr/local/bin/cosign

# バージョン確認
cosign version
02
GitHub Actions でキーレス署名（OIDC使用）

GitHub Actions環境では秘密鍵不要のキーレス署名が使えます。 GitHub OIDCトークンを使ってSigstoreが一時的な証明書を発行します。

.github/workflows/sign-and-push.yml 
name: Build, Sign & Push Container Image
on:
  push:
    tags: ['v*']

permissions:
  contents: read
  packages: write
  id-token: write   # ← OIDC トークン取得に必要

jobs:
  build-sign:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build & Push Container Image
        id: build
        uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/my-org/my-app:${{ github.ref_name }}

      - name: Install Cosign
        uses: sigstore/cosign-installer@v3

      - name: Sign Container Image (Keyless)
        run: |
          cosign sign \
            --yes \
            ghcr.io/my-org/my-app@${{ steps.build.outputs.digest }}
          # ↑ --yes でキーレス署名（OIDC認証）を使用

Kubernetes でのAdmission Control（Policy Enforcement）

Sigstore Policyコントローラーを使えば「署名されていないイメージのデプロイを自動ブロック」できます。

cosign-policy.yaml — 署名検証ポリシー 
apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-signed-images
spec:
  images:
    - glob: "ghcr.io/my-org/**"  # 組織のイメージすべてに適用
  authorities:
    - keyless:
        url: https://fulcio.sigstore.dev
        identities:
          - issuer: https://token.actions.githubusercontent.com
            subjectRegExp: "https://github.com/my-org/.*"
            # ↑ my-org のGitHub Actionsからのみデプロイを許可

Section 07 — EU Cyber Resilience Act

EU サイバーレジリエンス法（CRA）
─ セキュリティが法的義務になる時代

2024年12月発効。2026年9月から重大なインシデント報告が義務化。EU域内で販売するすべてのデジタル製品が対象。非準拠で最大1500万ユーロの罰金。

法的義務 2026年9月 — 報告義務開始 EU販売者全員対象
⚖️
EU CRAとは — なぜ日本企業も無視できないのか EU Cyber Resilience Act（CRA）は、EU域内で販売・提供される 「デジタル要素を含む製品」すべてにサイバーセキュリティ要件を法的に義務付ける規制です。 日本企業がEUに向けてソフトウェアやIoT機器を販売・SaaSを提供する場合も対象となります。 違反した場合は最大1500万ユーロ（または年間売上高2.5%）の罰金が科せられます。

CRA コンプライアンスタイムライン

2024年
12月10日
✅ CRA 正式発効
EUサイバーレジリエンス法が正式に官報掲載・発効。移行期間スタート。すべての製造業者・販売業者に準備を開始することが強く推奨される。
2026年
6月
⚡ 適合性評価機関の通知枠組み開始
適合性評価機関（認証機関）に関する法的枠組みが適用開始。製品カテゴリ別の適合性評価プロセスが始動する。今すぐ対応を開始する必要がある。
2026年
9月11日
🚨 最重要 — 脆弱性・インシデント報告義務開始
悪用された脆弱性および重大インシデントのENISA（欧州ネットワーク情報セキュリティ庁）への早期報告義務が開始。 これは2027年以前にリリースされた既存製品にも適用されるため、 現在販売中のすべての製品が対象です。報告期限は24〜72時間以内。
2027年
12月11日
📋 全要件の完全適用
すべてのCRA要件が完全適用開始。未準拠製品のEU市場での販売が禁止リスク。セキュリティ要件・SBOM提供・脆弱性対応のすべてが法的義務化。

CRAが要求する主要事項

要件 内容 対応ツール・方法
製品セキュリティ要件 既知の脆弱性がない状態でリリース。デフォルト設定が安全。認証・暗号化が適切 SAST/SCA/DAST by CI/CD
脆弱性対応義務 製品のサポート期間全体にわたって脆弱性修正を提供。修正は無償で提供 SBOM + 継続的CVEモニタリング
SBOM提供 機械可読なSBOMの作成・維持・提供（CycloneDX/SPDX形式） CycloneDX・Trivy・Syft
インシデント報告 悪用された脆弱性を24時間以内・重大インシデントを72時間以内にENISAへ報告 インシデント対応計画の策定
技術文書 セキュリティリスク評価・設計文書・テスト記録の保持（最低10年） 脅威モデリング文書の維持
CEマーキング クリティカルカテゴリ製品は第三者機関による適合性評価が必要 認定試験機関による評価

Section 08 — Tool Guide

2026年サプライチェーン
セキュリティツール完全ガイド

カテゴリ別に整理した推奨ツール。初学者が最初に導入すべきものを厳選して解説。

SBOM生成・スキャン

Trivy
OSS | 多機能
Aqua Securityが開発。コンテナ・ファイルシステム・IaC・SBOMの生成と脆弱性スキャンを1ツールで実現。CycloneDX・SPDX形式でSBOM出力可能。CI/CD統合が最も容易。まず最初に試すべきツール。
Syft
OSS | SBOM専門
Anchoreが開発するSBOM生成専用ツール。50以上のエコシステムに対応。CycloneDX・SPDX・Syft独自形式での出力。Grydeと組み合わせて脆弱性スキャンも可能。
Grype
OSS | 脆弱性DB
SBOMを入力として脆弱性スキャンを行うツール。NVD・GitHub Advisory Database・OSVなど複数のデータベースを参照。SyftとのパイプラインでSBOM生成→スキャンを一気通貫で実行。

署名・検証（Sigstore）

Cosign
OSS | 署名標準
コンテナイメージ・SBOM・バイナリへの署名と検証。キーレス署名（OIDC）に対応し秘密鍵管理が不要。GitHub Actionsとのネイティブ統合あり。2026年のデファクトスタンダード。
Rekor
OSS | 透明性ログ
Sigstoreの透明性ログシステム。すべての署名イベントを改ざん不可能な形で記録。Cosignによる署名は自動的にRekorに記録され、後から検証・監査が可能。
SLSA Verifier
OSS | 来歴検証
SLSA来歴（Provenance）の検証専用ツール。「このビルド成果物が指定したリポジトリ・ブランチから作られたか」を暗号学的に検証。デプロイパイプラインへの統合が推奨。

依存関係スキャン（SCA）

Dependabot
GitHub統合
GitHub標準搭載のSCAツール。脆弱な依存関係を検出してパッチ適用のPRを自動作成。GHSA・NVDデータベースを参照。有効化するだけで基本的なSCAが自動実行される。
Socket.dev
マルウェア特化
OSS依存関係に特化したSCA。Slopsquatting・タイポスクワッティング・悪意あるパッケージの検出に強み。AIコーディングツールが提案するパッケージの安全性確認に特に有効。
Snyk
商用 | 総合SCA
開発者フレンドリーな総合SCAtプラットフォーム。IDE統合でリアルタイムフィードバック。「到達可能性（Reachability）分析」で実際に悪用可能なCVEを優先表示しノイズを削減。

総括

サプライチェーンセキュリティは
「連鎖全体の強度」で決まる

チェーンの強度は最も弱いリンクで決まります。 どれだけ自社のコードが安全でも、依存するOSSライブラリ・CI/CDパイプライン・ コンテナイメージのいずれかに脆弱性があれば、それがそのまま組織のリスクになります。 SBOMで「何が入っているか」を可視化し、 SLSAで「どのように作られたか」を証明し、 Cosignで「誰が署名したか」を検証する ── この三位一体がサプライチェーンセキュリティの核心です。

ツール/標準 解決する問題 今すぐ始めるべき理由
SBOM 「何が入っているか」の可視化 EU CRA 2026年9月義務化・Log4Shellの教訓
SLSA 「どう作られたか」の証明 SolarWindス型CI/CD侵害を防ぐ唯一の方法
VEX 「本当に危険か」のフィルタリング SBOMが生むノイズを信号に変える
Cosign 「誰が作ったか」の署名検証 無料・キーレス・CI/CD完全統合可能
